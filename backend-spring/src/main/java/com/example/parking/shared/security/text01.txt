SecurityContextHolder là nơi Spring Security lưu thông tin user đang đăng nhập trong request hiện tại
nói dễ hiểu:
    - Sau khi JWT Filter kiểm tra token hợp lệ , nó cần báo cho Spring Security biết :
        + request này của user nào  , role là gì ?
        và thông tin đó được nhét vào SecurityContextHolder
Theo tài liệu Spring Security, SecurityContextHolder là nơi lưu chi tiết "ai được xác thực" và Spring Security sẽ đc dùng thông tin đó để Authorization
Ví dụ
java
```
    UsernamePasswordAuthenticationToken authentication =
    new UsernamePasswordAuthenticationToken(
        username,
        null,
        authorities
    );
    SecurityContextHolder.getContext().setAuthentication(authentication);
```
thì SecurityContextHolder là cái ngăn chứa bảo mật , đặt thông tin user đã đăng nhập vào đó
Sau khi set xong thì Spring security mới biết user hiện tại là ai vai trò j trạng thái Auth
    - Từ đó các Api mới kiểm tra đc @PreAuthorize("hasRole('STAFF')")
    hoặc .requestMatchers("/api/parking-session/**").hasRole("STAFF")


Controller service ; Tư duy hệ thống 
Mặc định trong Spring , tất cả các Bean @Controller , @ Service , @Repository đều đc khởi tạo dưới dạng 
Singleton Scope
    - Nghĩa là khi ứng dụng spring boot startup , nó chỉ tạo đúng 1 instance( đối tương) duy nhất
    cho mỗi class và lưu vào Spring Container để quản lý
    - Vì vậy cho dù hệ thống có nhận hàng triệu request , thì tất cả các thread xử lý đều gọi chung 1 instance đuy nhất này để thực thi hàm trong nó
Câu hỏi : Vậy tại sao dữ liệu của các Request ko bị đá nhau ?
( đá nhau ở đây là request A đang ghi dữ liệu của nó vào dữ liệu của object , và lúc đó thk request B lại cùng lúc lấy cái dữ liệu của thk A đi xử lý => bất ổn)
    - Nếu hàng trăm thread (request) cùng nhảy vào 1 hàm của instance Service duy nhất , 
    tại sao thông tin của User A không ghi đè lên User B ?

    - Vì Service chuẩn trong Spring thường là stateless
    - Tức là ko lưu trạng thái riêng của từng request vào filed
Ví dụ đúng :
@Service
public class ParkingSessionService {

    private final IParkingSessionRepositoryPort parkingSessionRepositoryPort;
    private final IVehicleRepositoryPort vehicleRepositoryPort; // đây là filed dependency sài ổn dùng để khởi tạo đúng 1 lần lúc chạy hàm constructor
    // Còn nếu có 1 hàm thay đổi data member trong quá trình chạy => nguy hiểm 

    public ParkingSessionService(
            IParkingSessionRepositoryPort parkingSessionRepositoryPort,
            IVehicleRepositoryPort vehicleRepositoryPort
    ) {
        this.parkingSessionRepositoryPort = parkingSessionRepositoryPort;
        this.vehicleRepositoryPort = vehicleRepositoryPort;
    }

    public void checkIn(String plateNumber) {
        String currentPlateNumber = plateNumber;

        // xử lý bằng biến local
        System.out.println("Đang xử lý xe: " + currentPlateNumber);
    }
}
